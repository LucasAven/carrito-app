# Siri entry creation via Shortcuts and a token-authed RPC

To let an Operator create an Entry by voice ("Hey Siri, nueva venta"), we expose a small POST endpoint that an iOS Shortcut calls, authenticated by a per-Operator personal token, with the privileged insert done inside a `security definer` Postgres function so the Next runtime never holds a `service_role` key. A web app (even an installed PWA) cannot register native Siri voice commands: WebKit exposes no App Intents / SiriKit API to web content, so the only voice path on iOS is the Shortcuts app, which triggers by phrase and can make an HTTP request to a backend. A Shortcut cannot carry the Supabase SSR session cookie, and Supabase access tokens expire hourly with rotating refresh tokens, so reusing the normal cookie/JWT auth is unworkable for a static shortcut; a long-lived, revocable personal token is the right primitive. We keep tenancy enforcement in a `security definer` RPC that hashes the presented token, resolves the owning `operator_id` from a new `api_tokens` table, and inserts the Entry with that `user_id`, rather than handing the Next app a `service_role` key: this keeps the privileged surface to one narrow function next to the data, preserves the multi-tenant guarantee from [ADR-0002](./0002-multi-tenant-from-day-one.md), and means a leaked app env never grants blanket table access. Tokens are shown once, stored only as a SHA-256 hash, and revocable via `revoked_at`. We rejected a native shell (Capacitor/SiriKit) as far too heavy for one feature, deep-link-into-PWA shortcuts as clunky and dependent on a live Safari session, and the in-app Web Speech API as unreliable on iOS and not actually Siri. The endpoint defaults a missing date to today computed in `America/Argentina/Buenos_Aires` (not server-UTC), consistent with how the app defines "today". Revisit if we adopt a native shell for other reasons (real App Intents would then beat Shortcuts) or if Apple ships a web App Intents API.
