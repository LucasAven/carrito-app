# Data API roles get no default table grants; grant per surface

The Supabase project runs with "Automatically expose new tables" **off**, so none of the Data API roles (`anon`, `authenticated`, `service_role`) receive default privileges on a new table. Row Level Security gates which rows a query sees, but base table privileges are a separate, earlier gate: without them every query fails with `42501 permission denied for table`, before RLS is even consulted. We grant exactly what each surface needs, explicitly, in migrations: `authenticated` gets read plus write on `entries`, `operators`, and `api_tokens` (see [`20260619051250_grant_data_api_access.sql`](../../supabase/migrations/20260619051250_grant_data_api_access.sql) and [`20260619060000_api_tokens.sql`](../../supabase/migrations/20260619060000_api_tokens.sql)); `anon` gets only `execute` on the `create_entry_via_token` SECURITY DEFINER function, never direct table access. `service_role` is granted **nothing** on these tables, by design. The consequence that bit us once, worth remembering: the **service-role key bypasses RLS but not table grants**, so a server-side script using it still gets `42501` until the privilege is granted. The one-time [legacy import](../legacy-import-plan.md) needed `select, insert on entries`; we granted it in the SQL editor for the run and revoked it after, deliberately **not** adding a tracked migration, so `service_role`'s standing surface stays empty (a one-time job should not widen a permanent grant). We rejected flipping "Automatically expose new tables" on, which would default-grant `anon`/`authenticated` to every future table and invert the least-privilege posture (a new table is private until a migration says otherwise). Revisit the transient-grant approach only if service-role scripting becomes routine enough to justify a narrow, permanent, well-documented grant.
