# New entries default to the viewed day, not today

The create drawers (Nueva Venta / Nuevo Gasto) now seed a new Entry's Occurred-on with the day the Operator is currently viewing in the DatePicker, falling back to today only in week/month view (where there is no single day). Previously they hardcoded today's date regardless of what was on screen, which was actively wrong for this app: retroactive entry is the normal workflow (see [CONTEXT](../../CONTEXT.md) and [ADR-0001](./0001-always-online.md), where books are filled end-of-shift, often catching up on past days), so the common act of navigating to a past day to backfill it forced a manual date change every single time. Worse, because `revalidatePath("/balance")` keeps the Operator on the day they were viewing, an entry stamped "today" while viewing the past silently vanished from view, reading as "it didn't save". Defaulting to the viewed day makes creation WYSIWYG (the new row appears immediately) and removes the friction. We rejected adding a backdating safeguard (a caption or accent when the date is not today): the Occurred-on field is always visible and one tap to change, and the dominant intent is backfill, so a guard would nag the common case to protect the rare one. We also rejected clamping the week/month-view default into the period, since creation from those views is rare and plain "today" is the safe, predictable choice. Each open re-seeds a fresh form so the date tracks navigation and stale half-entries do not linger. Revisit if logging shifts to live per-sale entry on the truck (today would become the better default), which would also reopen [ADR-0001](./0001-always-online.md).
